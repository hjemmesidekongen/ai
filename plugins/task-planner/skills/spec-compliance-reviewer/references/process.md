# Spec Compliance Reviewer — Full Process

Stage 1 of two-stage verification. All checks here are mechanical — no
judgment calls, no content quality assessment. That's Stage 2's job.

## When This Runs

The verification-runner calls this skill **before** any quality review.
If this skill returns `status: fail`, the verification-runner:
1. Marks the skill as `failed_spec` in state.yml
2. Skips Stage 2 (quality review) entirely
3. Routes failures back for structural fixes

## Input

From the target skill's SKILL.md frontmatter:
- `writes` — list of files/sections the skill declares it produces
- `checkpoint` — the checkpoint type and required checks

From the execution context:
- `state.yml` — current phase, skill status
- File-ownership map — from the plan's wave decomposition
- Working directory — where output files should exist

## Check 1: File Existence

Verifies every file declared in the skill's `writes` frontmatter exists
on disk.

**Procedure:**

1. Parse the target skill's SKILL.md frontmatter to extract `writes` list.
2. For each entry in `writes`:
   - If it's a file path (e.g., `brand-manual.md`): check the file exists
     at the expected location in the working directory.
   - If it's a section reference (e.g., `brand-reference.yml#colors`): check
     the parent file exists. Section content is checked in Check 2.
   - If it's a glob (e.g., `assets/**/*.svg`): use Glob to find matches;
     at least one match is required.
3. Record each expected file as present or missing.

**Result:**
```yaml
file_existence:
  status: pass | fail
  expected:
    - "brand-reference.yml"
    - "brand-manual.md"
  missing:
    - "brand-manual.md"    # only if missing
```

**Fail condition:** Any expected file is missing.

## Check 2: Schema Presence

For files that contain structured data (YAML), verifies the expected
sections and required top-level keys are present.

**Procedure:**

1. Identify which `writes` entries reference YAML sections (format:
   `filename.yml#section`).
2. Read each YAML file.
3. For each section reference:
   - Verify the top-level key exists in the YAML (e.g., `colors` key
     exists in `brand-reference.yml`).
   - Verify required sub-keys are present. Determine required sub-keys from:
     a. The skill's `checkpoint.required_checks` in frontmatter
     b. The check-templates.md for this checkpoint type
     c. The schema template in `resources/templates/` if available
4. For non-YAML files (markdown, HTML, shell scripts): skip this check
   and report `status: not_applicable`.

**Result:**
```yaml
schema_presence:
  status: pass | fail | not_applicable
  sections:
    colors:
      present: true
      required_keys: [primary, secondary, neutral, semantic]
      missing_keys: [semantic]
    typography:
      present: true
      required_keys: [fonts, type_scale, weights]
      missing_keys: []
```

**Fail condition:** Any expected section or required key is missing.

## Check 3: Non-Empty

Verifies no output is a stub, placeholder, or empty.

**Procedure:**

1. For each file in the `writes` list:
   - Check file size is > 0 bytes.
   - For YAML files: parse and verify no top-level section is `null`,
     empty string `""`, empty array `[]`, or empty object `{}`.
   - For YAML sections: verify at least one nested value is a non-empty
     string, number, or populated collection.
   - For markdown files: verify the file has at least one non-blank,
     non-heading line of content.
   - For findings.md: verify entries contain substantive text, not just
     template placeholders like "TODO" or "TBD".

2. Flag any file or section that appears to be a stub.

**Result:**
```yaml
non_empty:
  status: pass | fail
  empty_files:
    - file: "brand-reference.yml#colors"
      reason: "Section exists but all values are null"
```

**Fail condition:** Any output file is 0 bytes, or any YAML section
contains only null/empty values.

## Check 4: File Ownership

Verifies the skill only wrote to files it owns according to the
file-ownership map.

**Procedure:**

1. Read the file-ownership map from the plan (generated by the
   file-ownership skill during wave decomposition).
2. Get the list of files actually written during this skill's execution.
   Sources:
   - The skill's `writes` frontmatter (declared)
   - state.yml `phases[].artifacts` (actual, if logged)
   - Git diff since skill started (if available)
3. Cross-reference: every written file must appear in the ownership map
   as owned by this skill (or by the current wave).
4. Flag any file written that belongs to a different skill or wave.

**Result:**
```yaml
file_ownership:
  status: pass | fail
  violations:
    - file: "brand-reference.yml#voice"
      owner: "tone-of-voice"
      written_by: "typography-color"
```

**Fail condition:** Any file written by the skill is owned by a different
skill in the file-ownership map.

**Note:** If no file-ownership map exists (single-agent execution, no
parallel waves), this check returns `pass` with a note: "No ownership
map — single-agent mode."

## Check 5: State Consistency

Verifies state.yml reflects the skill's execution.

**Procedure:**

1. Read state.yml.
2. Find the phase entry for this skill (match by `name` field in
   `phases` array).
3. Verify:
   - An entry exists for this skill.
   - `status` is NOT `pending` (should be `in_progress`, `completed`,
     or a failure state).
   - If the skill ran successfully: status should be `in_progress` or
     `completed` (the verification-runner updates to `completed` after
     all checks pass — it's OK if it's still `in_progress` at this point).
   - `started_at` timestamp exists.
4. Check `current_phase` in state.yml matches this skill's name.

**Result:**
```yaml
state_consistency:
  status: pass | fail
  current_state: "in_progress"
  issues:
    - "Phase entry missing for skill 'typography-color'"
```

**Fail condition:** No phase entry exists, or status is still `pending`
after execution.

---

## Producing the Final Report

After running all 5 checks, combine results:

```yaml
spec_compliance:
  skill: [skill-name]
  ran_at: "[ISO timestamp]"
  status: pass | fail
  checks:
    file_existence: { ... }
    schema_presence: { ... }
    non_empty: { ... }
    file_ownership: { ... }
    state_consistency: { ... }
  failed_count: 0
  summary: "All 5 checks passed" | "2 of 5 checks failed: file_existence, non_empty"
```

**Overall status:**
- `pass` — all 5 checks passed. Stage 2 (quality review) can proceed.
- `fail` — one or more checks failed. Stage 2 is skipped. The skill is
  marked `failed_spec` in state.yml and failures are routed back for fixes.

## Error Handling

- **YAML parse failure:** If a YAML file can't be parsed, fail Check 2
  and Check 3 immediately with the parse error. Don't try to inspect
  individual sections.
- **Missing SKILL.md frontmatter:** If the target skill's SKILL.md has no
  `writes` field, report all checks as `not_applicable` except state
  consistency. Log a warning.
- **Missing state.yml:** Fail Check 5. Other checks can still run.

## Re-Run Behavior

When called after fixes:
1. Only re-run checks that previously failed.
2. Keep previous `pass` results.
3. Produce a new combined report with updated status.
