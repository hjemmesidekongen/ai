# Web Copywriting — Detailed Reference

## Copy Brief (Before Writing)

Before writing any copy, confirm:
1. **Page purpose**: type (homepage, landing, pricing, feature) + ONE primary CTA
2. **Audience**: target role, primary problem, what they've tried, main objections, their language
3. **Offer**: what's offered, key differentiator, primary outcome, available proof
4. **Context**: traffic source, awareness level (unaware -> problem-aware -> solution-aware -> product-aware)

## Above-Fold Templates

### SaaS Landing Page
```
Headline: [Specific outcome] without [main pain point]
Subheadline: [Product] helps [audience] [benefit] so they can [deeper outcome]
CTA: Start free trial | Get started free | See it in action
```

### Feature Page
```
Headline: [Feature name] that [outcome verb]s your [object]
Subheadline: [How it works in one sentence with a specific benefit]
CTA: Try [feature] free | See [feature] in action
```

### Pricing Page
```
Headline: Simple pricing for [audience type] of every size
Subheadline: [Risk reducer — free trial, no credit card, cancel anytime]
CTA: Start free | Choose plan | Get started
```

## CTA Formulas

Strong CTAs describe what the user gets, not what they do:
- "Start free trial" > "Sign up"
- "Get the report" > "Download"
- "See pricing" > "Learn more"
- "Create your first project" > "Get started"

CTA placement: above fold, after each major section, final section with risk reduction. Never more than one primary CTA visible at a time.

## Benefit Bridge Pattern

Every feature needs the "which means" bridge:

| Feature | Benefit Bridge | Outcome |
|---------|---------------|---------|
| AI-powered analytics | surfaces insights you'd miss manually | make better decisions in half the time |
| One-click deploy | skip the DevOps setup | ship features instead of configuring servers |
| Real-time collaboration | see changes as your team makes them | no more version conflicts or lost work |

## Specificity Upgrades

| Vague | Specific |
|-------|----------|
| Save time | Save 4 hours every week |
| Many customers | 2,847 teams |
| Fast results | Results in 14 days |
| Improve your workflow | Cut reporting time in half |
| Great support | Response within 2 hours |
| Easy to use | Set up in under 5 minutes |

If you can't make a claim specific, it's probably filler — cut it.

## Word-Level Editing

### Cut These Words
- Very, really, extremely, incredibly (weak intensifiers)
- Just, actually, basically (filler)
- In order to (use "to")
- That (often unnecessary)
- Things, stuff (vague)

### Replace These Words

| Weak | Strong |
|------|--------|
| Utilize | Use |
| Implement | Set up |
| Leverage | Use |
| Facilitate | Help |
| Innovative | New |
| Robust | Strong |
| Seamless | Smooth |
| Cutting-edge | Modern |
| Streamline | Speed up |
| Empower | Help |

### Sentence Rules
- One idea per sentence
- Vary sentence length (mix short and long)
- Front-load important information
- Max 25 words per sentence
- Active voice always ("Save your changes" not "Your changes will be saved")

### Paragraph Rules
- One topic per paragraph
- 2-4 sentences max for web
- Strong opening sentences
- Logical flow between paragraphs

## UI Text Patterns

### Error Messages
```
Structure: [What happened] + [Why, if helpful] + [What to do next]

Good: "Could not save changes. Check your connection and try again."
Bad:  "Error: Save failed."
Bad:  "An unexpected error has occurred. Please try again later."
```

### Empty States
```
Structure: [What would normally be here] + [Clear action to populate]

Good: "No projects yet. Create your first project to start tracking errors."
Bad:  "Nothing to display."
```

### Confirmation Dialogs
```
Structure: [Clear action in title] + [Consequences if destructive] + [Specific button labels]

Good: Title: "Delete this project?" Body: "This removes all data permanently." Buttons: "Delete project" / "Keep project"
Bad:  Title: "Are you sure?" Buttons: "OK" / "Cancel"
```

### Button Labels
- Action verbs: Save, Delete, Create, Send, Export
- Be specific: "Create Project" not "Create"
- Max 2-3 words
- No periods or exclamation marks

### Tooltips and Help Text
- Under 2 sentences
- Explain the "why", not just the "what"
- Link to docs for complex topics

## Seven Sweeps Checklist

### Sweep 1: Clarity
- [ ] Every sentence immediately understandable
- [ ] No jargon without explanation
- [ ] Pronouns have clear references
- [ ] No sentences trying to do too much

### Sweep 2: Voice and Tone
- [ ] Consistent formality level throughout
- [ ] Brand personality maintained
- [ ] No jarring mood shifts

### Sweep 3: So What
- [ ] Every feature connects to a benefit
- [ ] Claims answer "why should I care?"
- [ ] No impressive-but-empty statements

### Sweep 4: Prove It
- [ ] Claims substantiated with evidence
- [ ] Social proof is specific and attributed
- [ ] No unearned superlatives ("best", "leading", "#1")

### Sweep 5: Specificity
- [ ] Vague words replaced with concrete ones
- [ ] Numbers and timeframes included
- [ ] Filler content removed

### Sweep 6: Emotion
- [ ] Copy evokes feeling, not just information
- [ ] Pain points feel real and relatable
- [ ] Emotion serves the message authentically

### Sweep 7: Zero Risk
- [ ] Objections addressed near CTA
- [ ] Trust signals present (guarantee, trial, social proof)
- [ ] Next steps crystal clear

## Common Copy Problems

| Problem | Symptom | Fix |
|---------|---------|-----|
| Wall of features | List of capabilities without why | Add "which means..." benefit bridge |
| Corporate speak | "Leverage synergies to optimize" | Ask "how would a human say this?" |
| Weak opening | Starting with company history | Lead with reader's problem or desired outcome |
| Buried CTA | Ask comes after too much buildup | Make CTA obvious, early, and repeated |
| No proof | "Customers love us" without evidence | Add specific testimonials, numbers, cases |
| Generic claims | "We help businesses grow" | Specify who, how, and by how much |
| Mixed audiences | Speaks to everyone, resonates with none | Pick one audience per page section |

## Tone Decision: Plain Speech vs Brand Voice

| Use Plain Speech | Use Brand Voice |
|------------------|-----------------|
| Product UI (buttons, labels, forms) | 404 pages |
| Documentation | Empty states |
| Error messages | Onboarding flows |
| Settings pages | Loading states |
| Billing/payment flows | Marketing copy |
| Help text | "What's New" announcements |

Default to Plain Speech unless the context specifically calls for personality.
