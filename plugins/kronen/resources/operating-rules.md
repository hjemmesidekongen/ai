# Kronen Operating Rules

These rules are auto-distributed by the kronen plugin. They define behavioral contracts
that hooks enforce at runtime. Do not edit this file — it is the source template.
Changes here propagate to all projects via rules-sync.sh on SessionStart.

## Operating mindset

These principles apply to every task — freestyle, command, or plan-driven.

1. **Read before writing** — open the file, read the relevant code, understand the context. Never modify based on assumptions about what a file contains.
2. **Declare scope** — before executing, state what you'll change and why. If scope grows mid-task, pause and flag it.
3. **Check for existing process** — scan available skills and commands for a matching workflow before improvising. Adapt the process to the situation; don't ignore it.
4. **Define done** — state the success criteria before starting. Make each step verifiable.
5. **Minimum effective change** — write the least code that solves the problem. Resist the urge to refactor neighbors, add tests for unrelated code, or "improve" things not in scope.
6. **Surgical precision** — touch only what the task requires. If you spot unrelated issues, log them to `docs/opportunities.md` — do not fix them silently.
7. **Verify, don't infer** — run the proof command. Read the output. Never claim done based on "should work" or "looks right."
8. **Stop when done** — once the goal is met, stop. Don't add one more test, clean up one more file, or refactor one more function.
9. **Name confusion** — when something is ambiguous, say so. Present options. Push back. Don't guess and hope.
10. **Match effort to risk** — use junior-tier agents for low-risk tasks, senior for complex ones. Don't default to the most expensive option.

## Planning
For tasks involving 3+ files or multiple logical steps, plan before writing code.
For small, contained changes (single file, obvious fix), just do it — no plan needed.

**Routing decision tree:**
- IF task touches 1-2 files with obvious scope → just do it, no plan
- IF task touches 3+ files OR has multiple logical steps → load plan-engine skill
- IF user says "plan this", "break this down", "create a plan" → load plan-engine skill
- IF user runs `/plan` → load plan-engine skill via command
- IF user runs `/plan status` → read-only inspection, no planning skill needed
- IF an active plan exists (`.ai/plans/*/state.yml` with `status: in_progress`) AND user says "continue", "resume", "next" → load plan-engine skill, orient from existing state
- IF planning and no explicit `--manual` flag → default to autopilot execution

**Plan-engine is the only planning interface.** One skill, iterative-only. Goal fixed, path adapts.

## Verification and state
Always verify completed work before marking tasks done.
When working from a plan, update state.yml as work progresses (pending → in_progress → done).
Do not mark a plan complete until all waves pass verification.
If working ad-hoc (no plan file), no state tracking needed — but verification still applies.

## Agentic defaults
After completing a wave, always run verification before moving to the next wave.
If a task is blocked, surface the blocker and suggest alternatives — do not silently skip it.
