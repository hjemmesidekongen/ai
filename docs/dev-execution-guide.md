# Execution Guide: dev Plugin

## Reality Check

Building the dev plugin is an 18-step process. Each step builds one skill or
command, with a checkpoint at the end. You'll need approximately 18
sessions in Claude Code, doing one skill per session with `/compact` between.

**Rules:**
- One skill per Claude Code session
- `/compact` between sessions
- CLAUDE.md auto-updates after every step
- Git commit after every step
- Specs live in docs/ — Claude reads files, don't paste content
- If a step fails its checkpoint, fix it before moving on

## Prerequisites

Before starting, these spec documents must exist in docs/:
- `dev-implementation-plan.md` — generated by plugin-spec-generator
- `dev-addendum.md` — generated by plugin-spec-generator

### Step 1: Plugin scaffold + plugin.json

**Prompt:**

---

Read `plugins/task-planner/resources/plugin-blueprint.md` (Section 2: Required Plugin Structure).

Create the plugin scaffold:

1. Create directory structure:
   ```
   plugins/dev/
   ├── .claude-plugin/
   │   └── plugin.json
   ├── commands/
   ├── skills/
   ├── agents/
   ├── resources/
   │   ├── templates/
   │   └── examples/
   ├── scripts/
   └── README.md
   ```

2. Write `plugins/dev/.claude-plugin/plugin.json`:
   ```json
   {
     "name": "dev",
     "version": "1.0.0",
     "description": "Multi-agent development team with project-aware knowledge, framework packs, and 4-phase execution flow",
     "hooks": {
       "PreToolUse": [{ "matcher": "Write|Edit|Bash", "hooks": [{ "type": "command", "command": "cat state.yml 2>/dev/null | head -20 || true" }] }],
       "PostToolUse": [{ "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "echo 'dev: File updated. If this completes a phase, update state.yml.'" }] }],
       "SessionStart": [{ "hooks": [{ "type": "command", "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/session-recovery.sh" }] }],
       "Stop": [{ "hooks": [{ "type": "command", "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/check-wave-complete.sh" }] }]
     }
   }
   ```

   Write `plugins/dev/.claude-plugin/ecosystem.json`:
   ```json
   {
     "commands": ["dev:init", "dev:build", "dev:scan", "dev:status"],
     "skills": ["project-scanner", "config-generator", "knowledge-initializer", "feature-decomposer", "team-planner", "agent-dispatcher", "tier-assigner", "completion-gate", "code-review", "qa-validation", "delta-scanner"],
     "agents": ["project-manager", "software-architect", "product-owner", "frontend-tech-lead", "backend-tech-lead", "frontend-worker", "backend-worker", "mobile-developer", "code-reviewer", "qa-expert", "e2e-test-expert", "security-expert", "seo-expert", "devops", "designer"],
     "dependencies": ["task-planner"]
   }
   ```

3. Create `plugins/dev/scripts/session-recovery.sh` — see plugin-blueprint.md Section 13 for the template. Reads dev-config.yml and team-state.yml to report project context and current build status. Make executable with `chmod +x`.

4. Create `plugins/dev/scripts/check-wave-complete.sh` — see plugin-blueprint.md Section 13 for the template. Checks that the current skill or build phase is verified before allowing stop. Make executable with `chmod +x`.

5. Write `plugins/dev/README.md` with:
   - Plugin name: dev
   - Description: Multi-agent development team with project-aware knowledge, framework packs, and 4-phase execution flow
   - Installation: "This plugin is part of the claude-plugins ecosystem"
   - Commands: /dev:init, /dev:build, /dev:scan, /dev:status (with one-line descriptions)
   - Dependencies: task-planner (required), brand-guideline (optional enrichment)
   - Agent Roster: list the 15 agents with their roles and model tiers

Checkpoint type: file_validation
Required checks:
- plugin.json exists and is valid JSON
- plugin.json contains hooks (PreToolUse, PostToolUse, SessionStart, Stop)
- All directories exist (commands/, skills/, agents/, resources/templates/, resources/examples/, scripts/)
- scripts/session-recovery.sh exists and is executable
- scripts/check-wave-complete.sh exists and is executable
- README.md exists and lists all 4 commands
- ecosystem.json exists and is valid JSON
- ecosystem.json dependencies includes "task-planner"

Then update CLAUDE.md: check off this step in the Progress section and set "Next step" to Step 2. Commit everything.

### Step 3: Skill — project-scanner

> **What:** Static analysis — detect frameworks, conventions, and patterns in the project
> **Interactive:** No
> **Depends on:** None
> **Model tier:** Junior (Haiku)

**Prompt:**

---

Read `docs/dev-implementation-plan.md` (the "project-scanner" section under Skills).
Read `docs/dev-addendum.md` (the "Project Scanning and Framework Detection" section under Domain Knowledge).

Create `plugins/dev/skills/project-scanner/SKILL.md`
Also create `plugins/dev/skills/project-scanner/references/process.md` with the detailed execution steps (this skill will exceed 80 lines due to framework detection logic).

The SKILL.md must include these context engineering sections:

**Findings Persistence:**
During project scanning, write intermediate discoveries to the findings file:

`.ai/dev/[project-name]/findings.md`

**What to save:** detected frameworks with confidence levels, config file locations, ambiguous detections that need user confirmation

**2-Action Rule:** After every 2 file reads or Glob/Grep operations, IMMEDIATELY save key findings to findings.md before continuing.

**Format:**
```markdown
## Project Scan Findings
### Detected Frameworks
- [framework]: [version] (confidence: [high/medium/low], source: [file])
### Config Files Found
- [path]: [type]
### Ambiguous Detections
- [observation]: [possible interpretations]
```

**Error Logging:**
When errors occur during scanning (file read failures, parse errors, unexpected formats):
1. Log the error to state.yml errors array immediately
2. Before retrying any approach, check errors for previous failed attempts
3. Never repeat a failed approach — mutate strategy instead
4. The verification-runner logs checkpoint failures automatically

This skill:
1. Read package.json (if exists) at project root — extract `dependencies`, `devDependencies`, `scripts` sections
2. Glob for known config files: `next.config.*`, `vite.config.*`, `turbo.json`, `nx.json`, `prisma/schema.prisma`, `docker-compose.yml`, `Dockerfile`, `.github/workflows/*`
3. Detect language — check for `tsconfig.json` (TypeScript), `jsconfig.json` (JavaScript), `go.mod` (Go), `Cargo.toml` (Rust), `requirements.txt`/`pyproject.toml` (Python)
4. Detect package manager — check for `pnpm-lock.yaml` (pnpm), `yarn.lock` (yarn), `bun.lockb` (bun), `package-lock.json` (npm)
5. Detect architecture — check for `packages/` or `apps/` dirs (monorepo), `src/pages` or `src/app` (Next.js router), `src/routes` (SvelteKit/Remix)
6. Detect testing — check for `jest.config.*`, `vitest.config.*`, `playwright.config.*`, `cypress.config.*`
7. Detect linting/formatting — check for `.eslintrc.*`, `biome.json`, `.prettierrc.*`
8. Map src structure — identify entry points, key directories, component libraries (check for `components.json` for shadcn, material-ui imports)
9. Detect git conventions — read last 10 commit messages for patterns, check for `.github/` templates
10. Compile all detections into structured scan results object and save to findings.md

Output: internal scan results (passed to config-generator via findings.md, not written as standalone YAML)

Checkpoint type: data_validation
Required checks:
- Scan results contain at least 1 detected framework or explicit "none detected" marker
- Language field is populated (one of: typescript, javascript, python, go, rust, other)
- Package manager field is populated (one of: npm, yarn, pnpm, bun, unknown)
- At least 3 config file paths were checked (even if files were not found)
- findings.md exists with scan results

Commit Protocol (Subagent Mode):
When running as a subagent (dispatched via Task()):
1. Stage only files in your ownership list
2. Commit with message: `[plan_name]: project-scanner [task_id]`
3. Include commit SHA in your task_complete report
When running inline: commit after verification passes.

Verification (Two-Stage):
Stage 1 — Spec Compliance (Haiku):
  Run spec-compliance-reviewer against this skill's outputs.
  Checks file existence, schema presence, non-empty content, file ownership, state consistency.
  If FAIL: fix structural issues before proceeding. Do NOT run Stage 2.
Stage 2 — Quality Review (Opus):
  Only runs after Stage 1 passes.
  Run qa-reviewer against this skill's outputs for detection accuracy and coverage.
  If FAIL: address quality issues. If PASS_WITH_NOTES: review notes, decide whether to address.

Then update CLAUDE.md: check off this step in the Progress section and set "Next step" to Step 4. Commit everything.

### Step 6: Skill — tier-assigner

> **What:** Self-tiering logic for domain specialists — assess complexity at Haiku cost, auto-escalate on failure
> **Interactive:** No
> **Depends on:** None (utility skill)
> **Model tier:** Junior (Haiku)

**Prompt:**

---

Read `docs/dev-implementation-plan.md` (the "tier-assigner" section under Skills).
Read `docs/dev-addendum.md` (the "Model Tier Economics" section under Domain Knowledge).

Create `plugins/dev/skills/tier-assigner/SKILL.md`

This skill is concise enough for a single file (no references/process.md needed).

The SKILL.md must include these context engineering sections:

**Findings Persistence:**
During tier assessment, write assessment details to:

`.ai/dev/[project-name]/findings.md`

**What to save:** complexity factors assessed, tier decision reasoning, model floor checks

**2-Action Rule:** After every 2 assessment evaluations, IMMEDIATELY save findings to findings.md.

**Error Logging:**
When errors occur during tier assignment (invalid role, missing task brief):
1. Log the error to state.yml errors array immediately
2. Before retrying any approach, check errors for previous failed attempts
3. Never repeat a failed approach — mutate strategy instead
4. The verification-runner logs checkpoint failures automatically

This skill:
1. Receive task brief (description, scope, files_affected list) and agent role (from the agent roster in implementation plan)
2. Assess complexity factors:
   - Files affected: 1 file → junior, 2-5 files → senior, 6+ files → principal
   - Spec ambiguity: clear requirements → junior, some design choices → senior, unclear scope → principal
   - Risk level: isolated change → junior, feature-scoped → senior, critical path/system-wide → principal
   - Cross-cutting: no shared concerns → junior, touches shared utils → senior, architectural change → principal
3. Take the highest tier from the 4 factors as the baseline
4. Apply model floors: if agent is Code Reviewer or Security Expert, floor is senior (Sonnet). If agent is Designer, floor is senior (Sonnet). All others can tier down to junior (Haiku).
5. If a previous attempt at this task failed at a lower tier (check errors array in state.yml): auto-escalate one level above the failed tier
6. If already failed at principal (Opus): return `{ tier: "blocked", reason: "Failed at maximum tier" }` — PM must escalate to user
7. Return tier assignment: `{ tier: "junior"|"senior"|"principal", reasoning: "[1-2 sentences]", escalated_from: null|"junior"|"senior" }`

Output: tier assignment object (used by agent-dispatcher)

Checkpoint type: data_validation
Required checks:
- Tier is one of: junior, senior, principal, blocked
- Tier respects model floor for the agent role (code-reviewer, security-expert, designer ≥ senior)
- If escalated: previous tier and failure reason are recorded
- Reasoning is non-empty (at least 1 sentence explaining the assessment)

Commit Protocol (Subagent Mode):
When running as a subagent (dispatched via Task()):
1. Stage only files in your ownership list
2. Commit with message: `[plan_name]: tier-assigner [task_id]`
3. Include commit SHA in your task_complete report
When running inline: commit after verification passes.

Verification (Two-Stage):
Stage 1 — Spec Compliance (Haiku):
  Run spec-compliance-reviewer against this skill's outputs.
  If FAIL: fix structural issues before proceeding. Do NOT run Stage 2.
Stage 2 — Quality Review (Opus):
  Only runs after Stage 1 passes.
  Review tier logic correctness — floor enforcement, escalation paths, factor weighting.
  If FAIL: address quality issues. If PASS_WITH_NOTES: review notes, decide whether to address.

The SKILL.md must include these context engineering sections:

**Findings Persistence:**
During feature decomposition, write intermediate discoveries to:

`.ai/dev/[project-name]/findings.md`

This skill:
1. Receive the user's feature description (from /dev:build argument)

Output: writes `decomposition` section of `.ai/dev/[project-name]/team-state.yml`

Required checks:
- At least 1 component defined in `team-state.yml` `decomposition.components`

## Tips

1. **Read the spec, don't memorize it.** Every prompt tells you which file and section to read.
   Claude Code reads the file — you don't need to paste content.

2. **One skill per session.** Context degrades over long sessions. Build one skill,
   commit, `/compact`, start fresh.

3. **Checkpoints are gates.** If a checkpoint fails, the skill is not done.
   Fix the issues before moving to the next step.

4. **Build order matters.** Skills are ordered so each one can use the previous
   skill's output. Don't skip ahead.

5. **Interactive skills need the user.** If a skill is marked interactive,
   the user must be present to answer questions. Plan accordingly.

6. **Brand data is optional.** This plugin works without brand-guideline.
   If brand data is available, it enriches planning and design — but it's never required.

7. **Recovery works.** If a session crashes, check team-state.yml.
   It records exactly where you left off. Use `/dev:build --wave N` to continue.

8. **Agent roster is a catalog.** Not all 15 agents are used on every build.
   The PM selects the relevant agents based on the feature. A CLI tool might
   only need Backend Worker and DevOps — no Designer or Frontend Worker.
